## 一、文本切割：`smart_split_text()`

### 问题

原有 `load_text()` 只做了简单的按标点或按行分割，没有任何基于字符数的控制。一个长段落（如 500 字）会被整个送入模型，导致：

- 显存峰值急剧升高（长序列自回归生成需要更多 KV cache）
- 无法并行（每个任务大小差异极大，最慢的任务拖慢整体）
- 可能超过模型的 `max_text_tokens=600` 限制（来自 `config.yaml`）

### 解决方案

新增 `smart_split_text(text, min_chars=60, max_chars=100)` 函数，采用四级切割策略：

```
输入文本
    │
    ├─ 第1级：按句号/感叹号/问号/换行 → 自然句
    ├─ 第2级：累积句子，>min_chars 时输出
    ├─ 第3级：累计>max_chars 时在逗号/分号处折中切割
    └─ 第4级：单句>max_chars 强制在次级边界截断
```

- 优先在语义完整的句号处切割，保证 TTS 合成自然
- 长句不得已时在逗号处折中，避免生硬截断
- 支持中英文混合文本，仅统计 CJK 字符
- 尾段过短时自动合并到前一段

**效果举例**（一段 500 字文本）：
- 原方案：1 个段落 → 1 次 TTS 推理
- 新方案：5~8 段（每段 60~100 汉字）→ 可并行 3~4 段同时合成

## 二、半精度 FP16 自动开启

### 问题

原代码 `--fp16` 默认为 `False`，用户必须手工加参数才能开启。在 24GB+ 算力平台上，FP16 相比 FP32 可以：

- **节省 ~40% 显存**（模型权重减半）
- **提升 ~1.5~2x 推理速度**（内存带宽减半 + Tensor Core 加速）
- **几乎没有精度损失**（TTS 输出为音频，对低精度不敏感）

### 解决方案

```python
# 自动决策逻辑
if force_fp16 is not None:
```

## 三、并行 TTS 生成

### 问题

原代码串行逐段调用 `model.tts_to_file()`：

```python
for i, text in enumerate(segments):
    model.tts_to_file(text=text, ...)  # GPU 利用率低
```

即使在 24GB GPU 上，IndexTTS2 推理时的显存占用峰值约 **5.5GB (FP16) / 8GB (FP32)**，意味着显存大部分时间闲置。

### 解决方案

使用 `ThreadPoolExecutor` 实现多路并发：

```python
with ThreadPoolExecutor(max_workers=workers) as executor:
    futures = {executor.submit(_generate_one_segment, task): task[0]
               for task in tasks}
```

**显存自适应 worker 数计算**：

```
可用显存 = 总显存 - 基础开销(6GB)
workers  = floor(可用显存 / 每段峰值)

例：24GB GPU (FP16)
    可用 = 24 - 6 = 18 GB
    每段 = 5.5 GB
    workers = 18 / 5.5 = 3   → 3 路并行

例：32GB GPU (FP16)
    可用 = 32 - 6 = 26 GB
    workers = 26 / 5.5 = 4   → 4 路并行

例：48GB GPU (FP16)
    可用 = 48 - 6 = 42 GB
    workers = min(42/5.5, 8) → 8 路并行 (上限)
```

**加速预期**（以 100 段 TTS 为例）：

| 显存 | FP16 | Workers | 串行耗时 | 并行耗时 | 加速比 |
|:----:|:----:|:-------:|:--------:|:--------:|:------:|
| 24GB | ✅   | 3       | 500s     | ~200s    | **2.5x** |
| 32GB | ✅   | 4       | 500s     | ~150s    | **3.3x** |
| 48GB | ✅   | 8       | 500s     | ~80s     | **6.2x** |

## 四、WhisperX batch_size 自适应

### 问题

原代码 `batch_size=4` 固定写死。WhisperX 的 batch 参数控制每个 `transcribe()` 调用内同时处理的音频帧数，对 GPU 利用率影响很大。

### 解决方案

基于显存自动计算：

```python
whisper_usable = total_vram - 6.0    # 模型基础占用
batch_vram = 0.8 if fp16 else 1.5    # 每额外 batch 显存
max_batch = int(whisper_usable / batch_vram)
batch_size = clamp(max_batch, 1, 64)
```

典型值：

| 显存 | FP16 | 自动 batch_size | 相比原版 |
|:----:|:----:|:---------------:|:--------:|
| 24GB | ✅   | 16~22           | **4~5x** |
| 32GB | ✅   | 24~32           | **6~8x** |
| 48GB | ✅   | 48~64           | **12~16x** |
